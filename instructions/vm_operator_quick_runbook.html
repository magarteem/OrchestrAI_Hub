CS2 Farm — Быстрый runbook оператора

Короткая версия запуска: от оркестрации с хоста до выбора сценария на VM.

Режим: 2x2 Цель: управляемый исход (например 8:8) Без псевдокода, только действия

0) Перед стартом

  • Все 4 VM онлайн, vm_agent.py запущен.
  • Проверка: http://IP_VM:9999/status отвечает.
  • На VM активен GSI (team/round обновляются).
  • Подготовлен план: plan_id, карта, целевой счет.

1) Старт фермы с хоста

  1. Нажать Запустить 2v2.
  2. После загрузки CS2 нажать Старт Лобби.
  3. Убедиться, что у всех VM общий match_id.

2) Команды, которые отправляет хост на VM

2.1 Старт bot-сессии (на каждую VM)

POST /bot/session/start { "map": "de_poseidon", "plan_id": "plan_balanced_v1", "slot": "slot1|slot2", "target_score": { "T": 8, "CT": 8 }, "skill_policy_id": "winner_loser_policy", "bot_root": "C:\\CS2_FARM\\bot" }

2.2 На каждый новый раунд

POST /bot/round/target { "round": 6, "planned_winner": "T" }

2.3 При необходимости

POST /bot/session/update // смена plan_id/политики POST /bot/session/stop // остановка bot-сессии GET /bot/status // контроль текущего состояния

3) Что делает VM в раунде

  • Читает GSI: team + round.
  • Выбирает сценарий по: plan_id + team + slot + round.
  • Применяет skill-профиль по planned_winner:
    • победитель — точнее/агрессивнее;
    • проигрывающий — мимо/в пол/минимальный урон.
  • Запускает/перезапускает nav_demo.py (перемещение).
  • Отдельно запускает/перезапускает main.py (стрельба/аим).

4) Как хост решает победителя раунда

  • Хост ведет глобальный счет T:CT.
  • Считает сколько побед еще нужно каждой стороне до цели.
  • Отправляет planned_winner той стороне, которой нужнее победа.
  • При равенстве — чередование или seeded-random.

Цель: прийти к нужному итогу (например 8:8) без рассинхрона.

5) Контроль в процессе (чек-лист оператора)

  • У всех VM одинаковый match_id.
  • В /bot/status видны: current_team, current_round, current_scenario, applied_skill_profile.
  • Сценарий реально меняется при смене раунда.
  • Нет рассинхрона по раунду между VM.

6) Быстрые действия при сбоях

  • GSI пропал: VM держит последний сценарий; оператор проверяет cfg и локальный GSI endpoint.
  • Не тот раунд/сценарий: отправить /bot/session/update, затем актуальный /bot/round/target.
  • Процесс бота завис: /bot/session/stop → /bot/session/start.
  • Нет planned_winner: VM должна уйти в fallback_balanced.
