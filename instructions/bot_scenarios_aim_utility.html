Skill Model и Utility для VM Orchestrated

Обновленная схема для файла cs2_player_detection_demo/scenarios/de_poseidon_vm_runtime.json. Здесь описано, как применяется точность стрельбы по политике winner/loser в каждом раунде.

Ключевые блоки нового формата

  • scenario_catalog — маршруты сценариев (куда идти).
  • vm_selection.plans — ротация сценариев по team + slot + round.
  • skill_model.skill_profiles — профили стрельбы/точности.
  • skill_model.winner_loser_policy — правила, какой профиль дать победителю/проигрывающему.
  • skill_model.host_to_vm_fields — какие поля хост обязан отправлять VM.

Профили мастерства (skill_profiles)

  • idle_no_fire — почти без стрельбы.
  • lose_floor_spray — низкая точность, часто в пол/мимо.
  • lose_body_low — редкие попадания по телу.
  • balanced_body — средний сбалансированный профиль.
  • win_precise — высокая точность.
  • win_headshot_only — максимальная точность, приоритет headshot.

Числовой уровень задается полем aim_level внутри каждого профиля (0..5).

Как работает winner/loser policy

Хост на каждый раунд отправляет planned_winner (T или CT). VM по этому флагу выбирает набор skill-профилей для своей стороны и слота.

  • Если должен выиграть T, то T получает сильные профили, CT — слабые.
  • Если должен выиграть CT, то наоборот.
  • Если решения нет, VM использует fallback_balanced.

Что хост должен передавать на VM

  • На старте сессии: map, plan_id, slot, skill_policy_id.
  • На каждый новый раунд: planned_winner.

Utility в текущей версии

В de_poseidon_vm_runtime.json основной акцент на маршрутах и skill-профилях. Отдельный per-scenario блок utility_plan (как в старом формате) сейчас не выделен.

Если нужно, его можно добавить в каждый сценарий внутри scenario_catalog (например he_user и molotov_user).
