## Общая модель управления

- **Текущее состояние** хранится в глобальном `STATE` (`AgentState`).
- **GET**: только `/status`.
- **POST**: команды логина, запуска CS2, лобби-автоматизации, console tail.
- Большинство “долгих” операций запускаются в `daemon`-потоках и возвращают ответ сразу.

Состояния, которые реально используются:
- `idle`, `logging_in`, `logged_in`, `launching_cs2`, `cs2_running`, `opening_lobby_ui`, `lobby_ui_open`, `in_lobby`, `error`.

## Endpoint-карта

- `GET /status`
  - **Вход:** нет
  - **Предусловия:** нет
  - **Состояние:** не меняет
  - **Выход:** `state`, `steamid`, `login`, `error`, опционально `match_id` + `console_extra`

- `POST /login`
  - **Вход:** `login`, `password`, `shared_secret?`, `steam_path?`, `steam_launch_options?`
  - **Предусловия:** `STATE.state in ("idle","error")`
  - **Переходы:**  
    - сразу: `logging_in`  
    - потом (в потоке): `logged_in` при успехе, иначе `error`
  - **Риск:** можно отправить `/logout` или другие команды во время `logging_in`, получатся конфликтные переходы

- `POST /logout`
  - **Вход:** нет
  - **Предусловия:** нет
  - **Переходы:** принудительно `idle` (+ kill Steam)
  - **Риск:** ломает текущую фоновую операцию (логин/запуск), т.к. нет координации отмены

- `POST /cs2/launch`
  - **Вход:** `cs2_path`, `launch_options?`, `open_lobby_ui?`, `step_delay?`
  - **Предусловия:** `STATE.state == "logged_in"`
  - **Переходы:**  
    - сразу: `launching_cs2`  
    - в потоке: `cs2_running`  
    - если `open_lobby_ui=true`: `opening_lobby_ui` -> `lobby_ui_open`  
    - при ошибке: `error`
  - **Риск:** нет блокировки от повторного запуска, второй вызов может перетереть `_cs2_proc`/state

- `POST /cs2/kill`
  - **Вход:** нет
  - **Предусловия:** нет
  - **Переходы:** если состояние было `cs2_running|in_lobby|launching_cs2`, переводит в `logged_in`
  - **Риск:** при `opening_lobby_ui` не возвращает состояние явно; возможна несогласованность

- `POST /cs2/win_info`
  - **Вход:** нет
  - **Предусловия:** AutoIt доступен
  - **Состояние:** не меняет
  - **Назначение:** диагностика окна CS2

- `POST /lobby/create`
  - **Вход:** `steamids[]`, `shared_secret`, `password`
  - **Предусловия:** по state **не проверяет**
  - **Переходы:** при событии `logged_in` из `cs2_lobby.js` -> `in_lobby`
  - **Риск:** можно вызвать в неподходящем состоянии; слабая валидация сценария

- `POST /lobby/accept`
  - **Вход:** `lobby_id`, `cs2_cfg_path`
  - **Предусловия:** нет
  - **Переходы:** всегда ставит `in_lobby`
  - **Риск:** логическое состояние может стать `in_lobby`, даже если CS2 реально не готов

- `POST /lobby/create_ui`
  - **Вход:** `step_delay?`, `window_wait_timeout?`
  - **Предусловия:** `STATE.state in ("cs2_running","in_lobby")`, AutoIt
  - **Переходы:** `opening_lobby_ui` -> `lobby_ui_open` (или `error`)
  - **Риск:** повторный вызов во время уже идущей UI-последовательности

- `POST /lobby/get_friend_code`
  - **Вход:** `step_delay?`, `window_wait_timeout?`
  - **Предусловия:** state в `("idle","logged_in","cs2_running","in_lobby")`, AutoIt
  - **Состояние:** не меняет
  - **Выход:** `friend_code`
  - **Риск:** допускает `idle`, что не соответствует бизнес-логике (CS2 может быть не запущен)

- `POST /lobby/set_clipboard`
  - **Вход:** `code`
  - **Предусловия:** нет
  - **Состояние:** не меняет

- `POST /lobby/invite_by_code`
  - **Вход:** `friend_code`, `step_delay?`, `window_wait_timeout?`
  - **Предусловия:** state в `("idle","logged_in","cs2_running","in_lobby")`, AutoIt
  - **Состояние:** не меняет
  - **Риск:** те же допуски по состоянию (можно вызвать слишком рано)

- `POST /lobby/accept_invite`
  - **Вход:** `step_delay?`, `window_wait_timeout?`
  - **Предусловия:** только AutoIt
  - **Состояние:** не меняет

- `POST /lobby/start_match`
  - **Вход:** `window_wait_timeout?`
  - **Предусловия:** только AutoIt
  - **Состояние:** не меняет

- `POST /cs2/start_console_tail`
  - **Вход:** `cs2_path?`, `timeout_sec?`
  - **Предусловия:** нужен путь (`STATE.cs2_path` или из запроса)
  - **Состояние:** сбрасывает/обновляет `match_id`, `console_extra` в фоне
  - **Риск:** повторные старты останавливают предыдущий tail через общий `_tail_stop`, но без join/гарантированного завершения

- `POST /cs2/stop_console_tail`
  - **Вход:** нет
  - **Предусловия:** нет
  - **Состояние:** ставит `_tail_stop`

## Ключевые точки гонок и несогласованности

- **Глобальный `STATE` + много фоновых потоков**: переходы не сериализованы единой FSM-очередью.
- **Команды без строгих guard-условий** (`/lobby/*` часть) позволяют нелогичные вызовы.
- **`/logout` и `/cs2/kill`** могут пересечься с активным `/login` или `/cs2/launch`.
- **Смешение “фактического” и “логического” состояния**: некоторые endpoint-ы ставят `in_lobby` без подтверждения UI/игры.
- **Одинарный `HTTPServer`** (без auth и с глобальным состоянием) уязвим к внешним конфликтующим запросам.

## Как реализовать в твоем проекте

- **Шаг 1: добавить “точки внимания” (look-at points)** в сценарий рядом с маршрутными точками (`_client/_bot/scenarios/...json`).  
  Зачем: бот заранее знает опасные углы/выходы.

- **Шаг 2: в навигационном цикле держать 2 цели одновременно**:
  - `move_target` = следующая waypoint (движение),
  - `look_target` = опасная точка (взгляд).  
  Зачем: появляется полноценный стрейф/движение боком.

- **Шаг 3: yaw считать от `look_target`, а не от waypoint** в логике из `navigation/movement.py`.  
  Зачем: бот смотрит в угрозу, но продолжает идти по маршруту.

- **Шаг 4: WASD подавать в локальных координатах взгляда** (проекция вектора до waypoint на “forward/right” от текущего yaw).  
  Зачем: при взгляде вбок движение автоматически станет боковым (`A/D`), как у человека.

- **Шаг 5: переключать точки внимания по контексту** (дистанция до узла, сектор, видимость врага, таймер).  
  Зачем: не “залипает” на одном угле и выглядит естественно.

---

Мини-логика (ядро):

```python
move_vec = normalize(move_target - pos)
look_yaw = yaw_to(look_target - pos)       # куда смотреть
yaw_err = normalize_angle(look_yaw - yaw)

# камера/мышь
mouse_dx = clamp(yaw_err * k_p, -max_dx, max_dx)

# движение в локальной системе взгляда
forward = dot(move_vec, dir_from_yaw(yaw))
right   = dot(move_vec, right_from_yaw(yaw))

press_w = forward >  t
press_s = forward < -t
press_d = right   >  t
press_a = right   < -t
```

## Как получать “бег боком” корректно

Важно: если ты просто зажмешь `W`, а смотришь в сторону, траектория уйдет.

Нужен расчёт локального движения относительно текущего yaw:
- вычисляешь вектор до `move_target`;
- проецируешь его на локальные оси игрока (`forward/right`);
- по знакам/порогам жмешь комбинацию `W/A/S/D`.

Это даст естественный стрейф:
- бот может смотреть в опасный угол,
- но при этом реально идти по waypoint.

---

## Порядок обновления в тике (обязательно)

1. Определи `move_target` (next waypoint).
2. Определи `look_target` через `look_edges[team][from->to]`.
3. `desired_yaw = look_controller.resolve(...)`.
4. `mouse_dx = look_controller.compute_mouse_dx(current_yaw, desired_yaw)`.
5. Отправь `mouse_dx`.
6. Рассчитай WASD в локальной системе текущего yaw и отправь движение.

Так мышь и движение не конфликтуют.

---

## Практичные режимы look на ребре

Для каждой записи `from->to` поддержи:
- `fixed` — постоянный yaw (для “держать угол”),
- `point` — смотреть на мировую точку угрозы,
- `blend` — смешать направление в путь + danger угол (например, 30/70),
- `sweep` (опционально) — небольшой скан угла на длинных перебежках.

- dead-zone по yaw (не двигать мышь на мелких ошибках);

- если нет edge-профиля -> fallback “смотреть в сторону waypoint”.

**Cursor**

Данные хранятся в файле navmesh карты, в секции:

- `\_client\_bot\navmesh/<map>.json`
- ключ: `look_edges`

Формат такой:

```json
"look_edges": {
  "1->2": { "mode": "look_point", "x": 420.0, "y": -180.0 },
  "2->1": { "mode": "fixed_yaw", "yaw": 135.0 }
}
```

Зачем так:
- ключ `A->B` задает взгляд именно для движения из `A` в `B`;
- для обратного направления нужен отдельный ключ `B->A`.

Заполняется/редактируется через `navmesh_editor.py` в режиме `L (LOOK)`.

именно так и будет заполняться ?

  "2->1": { "mode": "fixed_yaw", "yaw": 135.0 }

что за ключи обекта  "1->2" и "2->1"

**Cursor**

Да, именно так будет заполняться.

- `"1->2"` и `"2->1"` — это **строковые ключи направленного ребра**:
  - `"1->2"` = когда бот идет **из waypoint 1 в waypoint 2**.
  - `"2->1"` = когда бот идет **из waypoint 2 в waypoint 1**.

Зачем это нужно:
- направления разные, поэтому и ключи разные;
- если задан только `"1->2"`, то для обратного хода `"2->1"` это правило не применится.

По содержимому:
- `mode: "look_point"` + `x,y` — смотреть в мировую точку.
- `mode: "fixed_yaw"` + `yaw` — держать фиксированный угол.

То есть эти ключи — просто идентификатор “откуда->куда” для конкретного участка маршрута.

Для маршрута `2->1->4->7` бот в каждый момент берет текущий сегмент:
- сначала `2->1`,
- потом `1->4`,
- потом `4->7`.

Поэтому в `look_edges` нужно задавать именно такие ключи:
- `"2->1"`, `"1->4"`, `"4->7"`.

Ключ `"2->7"` сработает **только если в пути реально есть прямой переход `2->7`** (как соседние точки в текущем шаге).  
Если бот идет через `1` и `4`, правило `"2->7"` в этой реализации не применяется.
